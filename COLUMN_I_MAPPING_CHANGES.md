# Изменения логики восстановления колонки I (артикулы)

## Commit: f16e371

## Проблема
При обновлении заказа 46206571-0647:
- Название товара изменилось: "Корпус Warrior Z5M" → "Корпус Warrior Z5"
- Старое значение колонки I: `4118275117-G1BZ` потерялось
- Автоматическое сопоставление через `difflib.get_close_matches()` не сработало (cutoff=0.6)

## Решение

### 1. Убрали автоматическое сопоставление
**Было**: 
- `difflib.get_close_matches(target_name, existing_names, n=5, cutoff=0.6)`
- Показывались только "похожие" названия (порог 0.6)
- Для Z5M vs Z5 порог не был достигнут → значение потеряно

**Стало**:
- Показываются **ВСЕ** доступные старые названия (первые 15)
- Пользователь выбирает вручную через Telegram
- Нет автоматических предположений

### 2. Улучшенный интерфейс запроса
```
🔄 Сопоставление колонки I

📦 Новое название: Корпус Warrior Z5

Доступные старые названия (со значениями I):
1. Корпус Warrior Z5M (2 знач.)
2. Материнская плата ASUS (1 знач.)
...

0. Пропустить (оставить I пустым)

💬 Выберите номер (0-15):
```

### 3. Отчет о потерянных значениях
После синхронизации выводится список потерянных значений:

```
✅ 🔄 Обновлено: 1
📝 Записано строк: 14

❗ Потеряно значений колонки I: 1
  • Корпус Warrior Z5M: 4118275117-G1BZ
```

### 4. Подтверждение сопоставления
После выбора варианта:
```
✅ Сопоставлено: Корпус Warrior Z5 → Корпус Warrior Z5M (2 знач.)
```

## Поведение

### Сценарий 1: Точное совпадение названия
- Значения восстанавливаются автоматически
- Запрос пользователю **НЕ** отправляется

### Сценарий 2: Название изменилось
- Отправляется запрос с **ВСЕМИ** доступными вариантами
- Пользователь выбирает номер (1-N) или 0 (пропустить)
- При выборе варианта: значения переносятся, отправляется подтверждение
- При пропуске: колонка I остается пустой

### Сценарий 3: Значения не восстановлены
- В конце синхронизации выводится отчет
- Формат: старое название + потерянное значение
- Пользователь может вручную найти и скопировать значения
